# Tanstack Query 입문 2편: useMutation과 Optimistic Update

## 어떤 상황에서 낙관적 업데이트(OptimisticUpdate)가 효율적일까요? 🍠

### 낙관적 업데이트(OptimisticUpdate)가 효율적인 상황은 언제일까요? 🍠

- **성공 확률이 높은 요청**: 좋아요, 팔로우, 댓글 작성처럼 실패할 가능성이 낮은 작업
- **빈번한 사용자 인터랙션**: 버튼을 여러 번 빠르게 누르는 경우처럼 즉각적인 피드백이 UX에 직결되는 상황
- **데이터 충돌 가능성이 낮은 경우**: 다른 사용자가 동시에 같은 데이터를 변경할 가능성이 적은 경우 (개인화된 데이터 등)
- **느린 네트워크 환경**: 응답 대기 시간이 길어 사용자가 버튼이 동작하지 않는다고 느낄 수 있는 상황

### 낙관적 업데이트(OptimisticUpdate)를 피해야 하는 상황 언제일까요? 🍠

- **실패 시 치명적인 작업**: 결제, 계좌 이체처럼 서버 응답을 반드시 확인해야 하는 경우
- **충돌 가능성이 높은 경우**: 여러 사용자가 동시에 같은 데이터를 수정할 수 있는 상황 (공동 편집 문서 등)
- **서버 측 유효성 검사에 의존하는 경우**: 클라이언트가 미리 알 수 없는 비즈니스 규칙이 있어 자주 실패할 수 있을 때
- **롤백이 사용자에게 혼란을 주는 경우**: 낙관적으로 추가된 데이터가 사라지는 것이 오히려 더 나쁜 UX를 만드는 상황

## 낙관적 업데이트(OptimisticUpdate) 블로그 읽고 개념 정리하기 🍠

### 낙관적 업데이트(OptimisticUpdate)를 왜 도입해야 하는지, 이 패턴이 해결하려는 문제를 실제 서비스 맥락에서 설명해보세요.

인스타그램 좋아요 버튼을 누를 때, 서버 응답을 기다렸다가 하트가 빨개진다면 어떨까? 네트워크 상황에 따라 0.5초~수 초의 딜레이가 생기고, 그동안 버튼이 반응하지 않는 것처럼 느껴져 사용자가 중복으로 누르게 된다.

낙관적 업데이트는 이 문제를 해결한다. 버튼을 누른 즉시 UI를 업데이트하고, 서버 요청은 백그라운드에서 처리한다. 실패하면 원래 상태로 롤백하고 사용자에게 알려주면 된다.

실제 서비스에서 해결하는 문제:

1. **체감 지연 제거**: API 응답을 기다리는 동안 발생하는 UI 멈춤 현상을 없앤다.
2. **중복 액션 방지**: 반응이 없다고 판단해 여러 번 클릭하는 행동을 줄인다.
3. **자연스러운 상호작용**: 앱보다 느리다는 인식 없이 네이티브 앱과 동일한 수준의 반응성을 제공한다.

### TanStack Query 기반 구현 흐름을 onMutate → (mutate) → onError → onSettled 순서로 기술해주세요.

**① onMutate** — 서버 요청 전에 실행

- `queryClient.cancelQueries()`: 진행 중인 refetch를 취소해 레이스 컨디션 방지
- `queryClient.getQueryData()`: 현재 캐시 데이터를 `previousData`로 저장 (롤백용)
- `queryClient.setQueryData()`: 캐시를 낙관적으로 즉시 업데이트 → UI가 바로 반응
- `return { previousData }`: context로 반환해 onError에서 사용할 수 있게 전달

**② mutate** — 실제 서버 API 호출 (TanStack Query가 내부적으로 처리)

**③ onError** — 서버 요청이 실패했을 때 실행

- `context.previousData`를 꺼내 `queryClient.setQueryData()`로 캐시를 원래대로 복원
- 필요 시 토스트 등으로 사용자에게 실패 알림

**④ onSettled** — 성공/실패 상관없이 항상 실행

- `queryClient.invalidateQueries()`: 해당 쿼리를 무효화해 서버 최신 데이터로 재동기화
- 낙관적 업데이트 결과와 서버 실제 결과를 일치시키는 마지막 단계

### ToDo “생성” 및 “좋아요 토글”에 낙관적 업데이트(OptimisticUpdate)를 적용했을 때의 실패/충돌 롤백 전략을 설계해주세요.

**ToDo 생성 롤백 전략**

- `onMutate`: 임시 id(예: `temp-${Date.now()}`)를 부여한 새 Todo를 목록 앞에 추가
- 실패 시 `onError`: `previousTodos`로 복원 → 임시 항목이 사라짐
- 성공 후 `onSettled`: `invalidateQueries`로 서버의 실제 id가 부여된 데이터로 교체
- **충돌 시나리오**: 임시 항목에 로딩 스피너 등 시각적 힌트를 표시해 “아직 저장 중”임을 사용자에게 알림

**좋아요 토글 롤백 전략**

- `onMutate`: `liked` 상태를 현재 값의 반대로 즉시 토글 + `previousLiked` 저장
- 실패 시 `onError`: `previousLiked`로 복원 → 원래 상태로 되돌아감
- **충돌 시나리오**: 사용자가 빠르게 여러 번 클릭하면 여러 요청이 동시에 발생할 수 있음
  - `cancelQueries`로 이전 요청을 취소하거나, 버튼에 debounce 적용
  - 또는 `onSettled`에서 항상 `invalidateQueries`로 서버 데이터와 동기화해 최종 상태를 보장
